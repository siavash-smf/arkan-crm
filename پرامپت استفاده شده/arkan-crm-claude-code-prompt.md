# پرامپت ساخت CRM آرکان (فاز ۴) — برای Claude Code

> این فایل را در ریشه‌ی پروژه‌ی «وب‌سایت + چت‌بات آرکان» (خروجی فازهای ۱ و ۲) قرار بده و متن زیر را به Claude Code بده. CRM روی همان پروژه‌ی Next.js و همان Supabase سوار می‌شود و نتیجه‌اش همین کدی است که در این ریپو می‌بینید.

---

## نقش و هدف

تو یک مهندس ارشد فول‌استک و متخصص طراحی سیستم‌های CRM هستی. قرار است برای شرکت مشاوره‌ی کسب‌وکار «آرکان» یک **CRM کامل با قابلیت‌های هوش مصنوعی** بسازی که در پنل مدیریت موجود (`/admin`) کنار بخش‌های چت‌بات بنشیند. این یک نمونه‌ی آموزشی کامل و حرفه‌ای است؛ پس کد باید تمیز، ماژولار و قابل‌توضیح باشد و از **best practiceهای شناخته‌شده‌ی صنعت CRM** (الگوی Salesforce/HubSpot) پیروی کند، نه یک طراحی من‌درآوردی.

**پیش از شروع:**
1. کد موجود را کامل بشناس: جدول `leads` الان صف ورودی مشترک دو کانال است (فرم مشاوره‌ی سایت و تولِ `capture_lead` چت‌بات که `conversation_id` هم ذخیره می‌کند). این جریان ورودی نباید بشکند.
2. قراردادهای پروژه را ادامه بده: دسترسی دیتابیس فقط سمت سرور با service-role و RLS بدون policy؛ اسکیما با فایل‌های SQL دستی در `supabase/`؛ تولید متن AI از طریق OpenRouter با مدل قابل‌تنظیم از دیتابیس؛ سرور‌اکشن‌ها به‌جای API route؛ UI فارسی RTL با توکن‌های برند (pine/brass/bone/sand) و اعداد فارسی.

**فلسفه‌ی معماری: «جریان کلاسیک CRM».** لید موجودیت خام ورودی است و دست‌نخورده می‌ماند؛ با اقدام آگاهانه‌ی کارشناس («تبدیل»)، به مخاطب (شخص) + شرکت (سازمان) + معامله (فرصت فروش) ارتقا پیدا می‌کند و از آن به بعد همه‌چیز — فعالیت‌ها، قرارداد، کمپین، هوش مصنوعی — حول پرونده‌ی مخاطب جمع می‌شود.

## مدل داده (Supabase)

این جداول را در یک فایل SQL جدید و idempotent بساز (جداول خالی `admin_users` و `audit_log` از فاز قبل موجودند و باید فعال/تکمیل شوند):

```
companies        (id, name, industry, website, city, size_label, notes, timestamps)
contacts         (id, company_id, full_name, phone, email, position,
                  source website|chatbot|manual, lead_id, conversation_id,
                  ai_summary, ai_summary_at, notes, timestamps)
pipeline_stages  (key PK, label_fa, position, is_won, is_lost)   ← مراحل config-driven، نه enum
deals            (id, title, contact_id, company_id, stage_key, status open|won|lost,
                  amount_toman bigint, expected_close, stage_entered_at,
                  won_at, lost_at, lost_reason, owner_email,
                  ai_next_action, ai_next_action_at, timestamps)
activities       (id, contact_id, deal_id, type call|meeting|note|task|stage_change,
                  title, body, due_at, done_at, created_by, created_at)
leads            (+ converted_at, contact_id, ai_score, ai_score_rationale, ai_scored_at)
admin_users      (+ password_hash, is_active, last_login_at)
audit_log        (+ actor_email, details jsonb)
contracts        (id, contract_no, title, deal_id, contact_id, company_id, body_md,
                  amount_toman, start_date, duration_label,
                  status draft|sent|viewed|accepted|canceled,
                  share_token, sent_at, viewed_at, accepted_at, accepted_by_name, …)
campaigns        (id, name, segment_key, goal, status, created_by, sent_at, created_at)
campaign_emails  (id, campaign_id, contact_id, lead_id, to_name, to_email,
                  context jsonb, subject, body_text,
                  status pending|ready|skipped|sent|failed, error, sent_at)
```

مراحل پایپ‌لاین را seed کن: جدید ← در حال بررسی ← جلسه مشاوره ← ارسال پروپوزال ← مذاکره ← بسته‌شده (موفق) ← بسته‌شده (ناموفق). وضعیت معامله باید **خودکار** از پرچم‌های `is_won`/`is_lost` مرحله تعیین شود.

## بخش اول: هسته‌ی CRM

### ۱. لیدها و تبدیل
- لیست موجود لیدها را نگه دار و به آن اضافه کن: دکمه‌ی **«تبدیل به مخاطب»** که شرکت را از نام کسب‌وکار find-or-create می‌کند (wildcardهای LIKE را escape کن)، مخاطب را با ارجاع به لید و گفتگوی چت‌بات می‌سازد، در صورت انتخاب یک معامله با مبلغ تخمینی می‌سازد، و متن چالش لید را به‌عنوان اولین یادداشت تایم‌لاین ثبت می‌کند. لید تبدیل‌شده نشان «تبدیل‌شده» بگیرد و دکمه‌اش به «مشاهده‌ی مخاطب» تبدیل شود.

### ۲. مخاطبان و شرکت‌ها
- CRUD کامل با جستجو؛ شرکت‌ها با شمارنده‌ی مخاطب/معامله.
- پرونده‌ی مخاطب = **نمای ۳۶۰ درجه**: کارت اطلاعات تماس + کارت «شناخت مشتری (AI)» + لیست معاملات با نشان مرحله + **تایم‌لاین یکپارچه** (ثبت لید، گفتگوی کامل چت‌بات به‌صورت بازشونده، فعالیت‌ها، رویدادهای تغییر مرحله) مرتب از جدید به قدیم.

### ۳. معاملات — برد کانبان
- ستون‌ها از جدول `pipeline_stages`؛ **درگ‌انددراپ با HTML5 بومی** (هیچ کتابخانه‌ی UI اضافه نکن) + fallback با select برای موبایل؛ جابه‌جایی optimistic با بازگشت در خطا.
- کارت: عنوان، مخاطب، مبلغ به تومان با ارقام فارسی، روزهای مانده در مرحله. فوتر ستون: جمع ارزش.
- انتقال به مرحله‌ی باخت، دلیل بگیرد؛ هر جابه‌جایی یک activity از نوع `stage_change` در تایم‌لاین مخاطب ثبت کند.

### ۴. فعالیت‌ها و وظایف
- تماس/جلسه/یادداشت/وظیفه با سررسید؛ فیلتر نوع و وضعیت؛ وظیفه‌ی **معوق قرمز** و «امروز» طلایی؛ چک‌باکس انجام.
- مقدار `datetime-local` را **سمت مرورگر** به ISO تبدیل کن تا تایم‌زون سرور (UTC روی Vercel) زمان را جابه‌جا نکند.
- کارت «وظایف معوق» به داشبورد اصلی اضافه شود.

### ۵. گزارش‌ها
- قیف تبدیل (لید ← تبدیل‌شده ← معامله ← جلسه به بعد ← برد با درصدها)، ارزش پایپ‌لاین به تفکیک مرحله، منابع لید (وب‌سایت/چت‌بات)، و درآمد ماهانه‌ی ۱۲ ماه اخیر با نام ماه‌های **تقویم فارسی**.
- تجمیع در TypeScript و رندر با همان الگوی نمودار div-bar داشبورد موجود — کتابخانه‌ی چارت اضافه نکن.

## بخش دوم: احراز هویت چندکاربره و ممیزی

- ورود با **ایمیل + رمز** از جدول `admin_users`؛ هش رمز با **scrypt از `node:crypto`** (هیچ کتابخانه‌ی auth اضافه نکن). کوکی نشست: payload امضاشده با HMAC شامل `{uid, email, role, exp}` با ۳۰ روز اعتبار.
- **پنج نقش**: owner > admin > editor > operator > viewer. بیننده در **کل پنل** (CRM و چت‌بات) فقط‌خواندنی است؛ ساخت/تنزل admin و owner فقط توسط owner؛ یک admin نتواند owner را غیرفعال کند و پنل قفل شود.
- **مسیر مهاجرت نرم**: تا وقتی هیچ کاربری با رمز ساخته نشده، رمز واحد قدیمی (`ADMIN_PASSWORD`) کار کند و صفحه‌ی ورود فرم «ایجاد اولین مالک» داشته باشد (رمز قدیمی = کد راه‌اندازی). بعد از ساخت اولین کاربر، مسیر قدیمی بسته شود — و **fail-closed**: خطای دیتابیس نباید مسیر قدیمی را دوباره باز کند. نشست‌های موجود ادمین فعلی هم نباید بپرند.
- صفحه‌ی «کاربران»: مدیریت نقش/فعال‌سازی/ریست رمز + جدول آخرین ردیف‌های **audit log**.
- helper ممیزی `logAudit` بساز که هر عملیات مهم (ورود، تبدیل لید، تغییر مرحله، عملیات AI و…) را با ایمیل عامل ثبت کند. نام عملیات‌ها را **نوع‌دار** کن (union) تا عملیات بدون لیبل فارسی خطای کامپایل بدهد. insertهای audit را **await** کن — روی serverless، درخواست fire-and-forget بعد از ارسال پاسخ ممکن است هرگز کامل نشود.

## بخش سوم: قابلیت‌های هوش مصنوعی

همه از همان OpenRouter و مدل فعال `model_config` استفاده کنند؛ خروجی‌های ساخت‌یافته با zod اعتبارسنجی و یک بار retry شوند؛ بدون کلید API، پیام خطای فارسی تمیز (نه crash).

1. **امتیازدهی لید**: دکمه روی هر کارت لید؛ نمره‌ی ۰–۱۰۰ + توضیح کوتاه فارسی از روی چالش، مرحله‌ی کسب‌وکار، کامل‌بودن اطلاعات و منبع؛ روی ستون‌های لید ذخیره و به‌صورت چیپ رنگی نمایش داده شود.
2. **خلاصه‌سازی گفتگوی چت‌بات**: در پرونده‌ی مخاطبِ دارای `conversation_id`؛ ۳–۵ بولت فارسی (نیازها، دغدغه‌ها، سیگنال خرید) در کارت «شناخت مشتری».
3. **پیشنهاد اقدام بعدی معامله**: از روی مرحله، مدت توقف، مبلغ و ۵ فعالیت آخر؛ روی کارت کانبان نمایش داده شود.
4. **چت با CRM** (صفحه‌ی «دستیار»): رابط چت استریمی که مدل با **tool-calling** به داده‌ی واقعی جواب بدهد — چهار ابزار با ورودی zod: `query_leads`، `query_deals`، `query_activities`، `crm_stats`. مدل حق ندارد از خودش عدد بسازد؛ پاسخ تحلیلی فارسی. فقط برای ادمین واردشده (حواست باشد کوکی نشست با `path=/admin` ست می‌شود؛ روت API باید زیر همان مسیر باشد).

> ⚠️ نکته‌ی فنی مهم: مدل فعال پروژه reasoning دارد (Gemini Flash) و بخش قابل‌توجهی از `maxOutputTokens` را صرف استدلال می‌کند؛ سقف توکن توابع AI را بالا بگیر (۲۰۰۰+) و صفحاتی که اکشن AI صدا می‌زنند `maxDuration = 60` بگیرند، وگرنه JSON نیمه‌کاره و timeout می‌گیری.

## بخش چهارم: قراردادها

- ساخت قرارداد از روی مخاطب/معامله با شماره‌ی خودکار (`AR-<سال شمسی>-<ترتیبی>`)؛ متن Markdown از **قالب استاندارد ۱۰ ماده‌ای قرارداد مشاوره‌ی آرکان** (طرفین، موضوع با متدولوژی «چهار رکن»، مدت، پرداخت مرحله‌ای ۴۰/۳۰/۳۰، تعهدات طرفین، محرمانگی، فسخ، حل اختلاف) با مقادیر واقعی کلاینت جایگذاری شود.
- ویرایشگر Markdown + دکمه‌ی **«بازنویسی با AI»** که متن را با چالش لید و شناخت AI مشتری شخصی‌سازی کند.
- **صفحه‌ی عمومی `/contract/[share_token]`** با سربرگ کامل برند (لوگو، رنگ‌ها، RTL): کلاینت بدون لاگین قرارداد را ببیند، **چاپ/ذخیره PDF** کند (print CSS؛ نوار اقدامات در چاپ حذف شود) و با واردکردن نامش **تأیید آنلاین** کند (امضای ساده با ثبت زمان).
- چرخه‌ی وضعیت زنده: پیش‌نویس ← ارسال‌شده (دکمه‌ی ارسال، لینک را کپی کند + گزینه‌ی mailto) ← **دیده‌شده** (اولین بازدید کلاینت خودکار ثبت شود) ← تأییدشده (قفل ویرایش) / لغوشده.
- **پیگیری هوشمند**: برای قرارداد ارسال/دیده‌شده‌ی تأییدنشده، دکمه‌ای که AI پیام پیگیری مؤدبانه‌ی فارسی تولید کند (با اشاره به روزهای انتظار و لینک) + کپی و ارسال با ایمیل.

## بخش پنجم: کمپین ایمیلی شخصی‌سازی‌شده (human-in-the-loop)

- **سگمنت‌های code-driven** (در یک ماژول جدا تا الگوی segment builder آموزش داده شود): لیدهای ازدست‌رفته، لیدهای پیگیری‌نشده‌ی ۷+ روز، معاملات باخته، مشتریان برنده — سقف ۲۰ گیرنده‌ی دارای ایمیل.
- ساخت کمپین = نام + سگمنت + «هدف» (که به AI داده می‌شود). برای هر گیرنده AI یک ایمیل **کاملاً اختصاصی** بنویسد: اشاره به چالش/وضعیت خودِ او، حداکثر ۱۲۰ کلمه، بدون لحن تبلیغاتی، با CTA جلسه‌ی رایگان.
- تولید **تک‌به‌تک** انجام شود (هر گیرنده یک درخواست) تا هیچ اکشنی به timeout نخورد؛ دکمه‌ی «تولید همه» سمت کلاینت حلقه بزند و پیشرفت نشان بدهد.
- ادمین قبل از ارسال هر ایمیل را ببیند/ویرایش/رد کند. ارسال با **Resend** (REST مستقیم، بدون SDK)؛ بدون `RESEND_API_KEY` همه‌چیز تا مرحله‌ی بازبینی کار کند و راهنمای فارسی نمایش داده شود.

## بخش ششم: پیوند با بلاگ (فاز ۳)

- دکمه‌ی **«پیشنهاد محتوا»** روی هر لید: AI از بین پست‌های منتشرشده‌ی بلاگ (از `listPublishedPosts` موجود) مرتبط‌ترین را با چالش لید انتخاب کند و پیام کوتاه پیگیری بنویسد؛ خروجی با لینک پست نمایش و قابل کپی باشد؛ اگر لید تبدیل شده، به‌عنوان یادداشت در تایم‌لاین مخاطب هم ثبت شود.

## ناوبری و UX

- ناوبری پنل را **گروهی** کن: داشبورد | CRM (لیدها، مخاطبان، شرکت‌ها، معاملات، فعالیت‌ها، قراردادها، کمپین‌ها، گزارش‌ها، دستیار) | چت‌بات | تنظیمات (ویجت، تلگرام، کاربران) — بدون تغییر URLهای موجود؛ تب «کاربران» فقط برای admin به بالا دیده شود.

## الزامات کیفی

- الگوی ثابت هر سرور‌اکشن: چک نشست و نقش ← اعتبارسنجی zod ← عملیات ← audit ← revalidate. پیام خطاها فارسی.
- fetchهای مستقل صفحات با `Promise.all` موازی شوند؛ کد تکراری UI (چیپ، اسپینر، فرمت تاریخ، کلاس input) در یک ماژول مشترک.
- `npm run build` باید سبز بماند و هیچ dependency جدیدی اضافه نشود.
- یک اسکریپت **داده‌ی نمایشی** (`scripts/seed-crm-demo.mjs`) بساز با کلاینت‌های خیالی ایرانی واقع‌گرایانه که همه‌ی بخش‌ها را برای دموی کلاس پر کند: ~۱۰ شرکت، ~۱۵ مخاطب، معاملات در همه‌ی مراحل + برد/باخت پخش‌شده در ماه‌های گذشته (برای نمودار درآمد)، فعالیت‌های معوق/امروز/آینده/انجام‌شده، چند لید تازه برای دموی زنده‌ی تبدیل، و یکی‌دو گفتگوی چت‌بات نمونه‌ی متصل به مخاطب — همه نشانه‌گذاری‌شده + سوییچ `--clean` برای پاک‌سازی.
- README را با مستندات کامل CRM (معماری، راه‌اندازی، ترتیب SQLها، سناریوی دمو) به‌روز کن.

## ترتیب کاری که از تو می‌خواهم

1. کد موجود را بررسی کن و خلاصه‌ی درکت از معماری فعلی را بده؛ اگر تصمیم محصولی مهمی باز است، اول از من بپرس.
2. فایل SQL اسکیمای CRM را بساز (من دستی در SQL Editor اجرا می‌کنم).
3. احراز هویت چندکاربره و audit — طوری که همه‌ی صفحات موجود بدون تغییر کار کنند.
4. ناوبری گروهی، سپس مخاطبان/شرکت‌ها/تبدیل لید، سپس کانبان معاملات، سپس فعالیت‌ها.
5. قابلیت‌های AI، سپس گزارش‌ها، سپس قراردادها، کمپین‌ها و دستیار.
6. اسکریپت داده‌ی نمایشی، تست end-to-end (از فرم سایت تا برد معامله و گزارش درآمد)، بازبینی امنیتی و build نهایی.

در هر مرحله مختصر توضیح بده چه ساختی و بعد ادامه بده.
